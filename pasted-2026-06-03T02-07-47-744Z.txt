## 1. Visão Geral e Missão

### 1.1 O Produto

TravelOS é uma plataforma SaaS multi-tenant para agências de viagens que unifica CRM, proposta comercial, contratação, operação de embarque, portal do cliente e financeiro em um único workspace minimalista e orientado por fluxo de trabalho.

### 1.2 Stakeholders

| Perfil | Papel |
|---|---|
| **Super Admin (Global)** | Gerencia todas as agências, planos, permissões e dados globais |
| **Admin da Agência** | Gerencia agentes, brand kit, financeiro e configurações da agência |
| **Agente de Viagens** | Usa CRM, cria propostas, gera contratos, faz embarques |
| **Cliente** | Acessa portal pessoal, assina contratos, acompanha viagens |
| **Viajante** | Preenche dados pessoais via magic link, faz check-in online |

### 1.3 Fluxo Macro do Produto

```
Lead Captado
    │
    ▼
CRM Kanban (Lead → Qualificado → Proposta → Fechado)
    │
    ▼
Cotação/Proposta (CMS + Canvas A4 + Web View)
    │
    ▼ Cliente aceita
Contrato Digital (Cláusulas Pétreas + Assinatura Eletrônica ICP-B)
    │
    ▼
Gestão de Passageiros (Magic Links por Quarto/Pax)
    │
    ▼
Kanban de Embarques (Checklists + Vouchers + Vistos)
    │
    ▼
Voucher Personalizado (OCR Operadora → AI → PDF/PNG 8K)
    │
    ▼
Portal do Cliente (Timeline, Pagamentos, Documentos)
    │
    ▼
Suporte/Ticket (Pós-Venda, Alterações, Email/WA)
    │
    ▼
Financeiro (DRE por Viagem + Caixa Geral)
```

## 3. Arquitetura Técnica

### 3.1 Stack

| Camada | Tecnologia |
|---|---|
| Frontend | React 18 + TypeScript + Vite |
| UI Framework | Tailwind CSS (config baseado nos tokens acima) |
| Componentes | shadcn/ui (customizado conforme design system) |
| Estado | Zustand (global) + React Query (server state) |
| Backend | Supabase (PostgreSQL + Auth + Storage + Realtime + Edge Functions) |
| IA | Google Gemini API (gemini-2.5-flash para OCR/texto; imagen-4.0 para imagens) |
| PDF | html2pdf.js + html2canvas (scale 4 = qualidade 8K) |
| Assinatura | signature_pad + hash SHA-256 + metadados ICP-B |
| Drag & Drop | @dnd-kit/sortable (Kanban) |
| Mapas | Leaflet + OpenStreetMap (Nominatim geocoding) |
| Gráficos | Recharts |
| Formulários | React Hook Form + Zod |
| Email | Resend API |
| WhatsApp | Meta WhatsApp Business Cloud API |
| MCP | @modelcontextprotocol/sdk |

### 3.2 Multi-tenancy

```
Estrutura de isolamento:
├── global_admin          → schema público, sem RLS
├── agencies              → tenant principal
│   ├── agents            → RLS: agent.agency_id = agency.id
│   ├── clients           → RLS: agency_id
│   ├── trips             → RLS: agency_id
│   ├── contracts         → RLS: agency_id (READ ONLY após assinatura)
│   ├── financial         → RLS: agency_id
│   └── ...
```

```

  - Razão social
  - CNPJ

Step 2: Responsável Legal
  - Nome completo
  - CPF

Step 3: Brand Kit Básico
```

## 12. Módulo: Contratos Digitais

Rota: `/agency/:slug/trips/:id/contract` e `/agency/:slug/contracts`

### 12.1 Princípio de Imutabilidade

> **REGRA ABSOLUTA:** Após assinatura, NENHUM campo do contrato pode ser alterado. O documento é selado com hash SHA-256 e todos os metadados são imutáveis.

### 12.2 Schema do Contrato

```typescript
interface Contract {
  id: string;
  agency_id: string;
  trip_id: string;
  // Documento
  version: string;                // "1.0.0"
  content_hash: string;           // SHA-256 do conteúdo pré-assinatura
  signed_hash: string;            // SHA-256 do conteúdo + metadados pós-assinatura
  pdf_url: string;                // URL imutável no Storage
  // Partes
  agency_data: ContractParty;     // snapshot da agência no momento
  client_data: ContractParty;     // snapshot do cliente no momento
  passengers_data: ContractPassenger[];
  // Conteúdo
  fixed_clauses: ContractClause[]; // 49 cláusulas pétreas (ver seção 29)
  custom_clauses: ContractClause[]; // cláusulas adicionais da agência
  package_summary: string;         // resumo do pacote
  total_value: number;
  payment_terms: string;
  // Assinaturas
  signatures: ContractSignature[];
  // Certificado
  certificate: ContractCertificate;
  // Status
  status: 'draft' | 'sent' | 'pending_signature' | 'signed' | 'cancelled';
  created_at: Date;
  signed_at?: Date;
  cancelled_at?: Date;
  cancellation_reason?: string;
}

interface ContractSignature {
  party: 'agency' | 'client' | 'passenger';
  party_id: string;
  full_name: string;
  cpf: string;
  // Dados de captura
  signature_image_base64: string; // imagem da assinatura desenhada
  selfie_image_base64?: string;   // foto tirada no momento da assinatura
  ip_address: string;
  user_agent: string;
  device_fingerprint: string;
  geolocation?: { lat: number; lng: number; city: string; };
  signed_at: Date;
  // Hash individual
  signature_hash: string;         // SHA-256(nome+cpf+ip+timestamp+conteudo)
}

interface ContractCertificate {
  serial: string;                  // UUID do certificado
  issued_at: Date;
  issuer: string;                  // "TravelOS Digital Certificate Authority"
  algorithm: 'SHA-256';
  chain_hash: string;              // hash encadeado de todas as assinaturas
  verification_url: string;        // URL pública para verificar autenticidade
  legal_text: string;              // declaração legal de validade
}
```

### 12.3 Fluxo de Assinatura

```
1. Agente gera contrato (pré-preenchido com dados da viagem)
2. Sistema gera PDF draft + calcula content_hash
3. Agente envia magic link para o cliente: /m/contract/:token
4. Cliente:
   a. Lê o contrato na íntegra (scroll obrigatório monitorado)
   b. Preenche dados pessoais
   c. Tira foto selfie (via câmera do dispositivo)
   d. Desenha assinatura (canvas signature_pad)
   e. Confirma geolocalização
   f. Clica "Assinar Contrato"
5. Sistema:
   a. Captura IP, user_agent, device_fingerprint
   b. Calcula signature_hash
   c. Atualiza signed_hash (encadeia todas as assinaturas)
   d. Gera PDF final imutável
   e. Salva em Supabase Storage (bucket protegido, sem delete)
   f. Registra timestamp em tabela de auditoria (append-only)
6. Ambas as partes recebem PDF por email
```

### 12.4 Cofre de Contratos

Acesso em: `/agency/:slug/contracts`

- Listagem com filtros: por cliente, data, status, valor
- Visualização do PDF no próprio sistema
- Página de verificação pública: `/verify/:certificate_serial`
- Exportação em lote (ZIP com PDFs)
- A agência pode criar um contrato avulso (sem proposta)

### 12.5 Cláusulas Pétreas (Ver Seção 29)

O sistema vem com 49 cláusulas pré-configuradas baseadas no modelo da Excelência Tour. A agência pode:
- **Adicionar** cláusulas adicionais
- **NÃO pode** remover ou alterar as 49 cláusulas base

```typescript
// Configuração padrão (IMUTÁVEL — qualidade máxima)
const PDF_CONFIG = {

  },
  jsPDF: {
    unit: 'mm',
    format: 'a4',

    compress: true,
  },
  pagebreak: { mode: ['css', 'legacy'] },

// Story/Story (voucher, passagem)

  height: 711,
```

## 29. Regras de Negócio e Cláusulas Pétreas

### 29.1 As 49 Cláusulas Pétreas

> Estas cláusulas são IMUTÁVEIS no sistema. Não podem ser removidas, alteradas ou reordenadas por nenhum usuário, incluindo o Super Admin Global. São a base legal de todos os contratos.

**Estrutura:**

```typescript
interface ContractClause {
  number: number;             // 1 a 49 (pétreas) ou 50+ (customizadas)
  section: string;            // "DO OBJETO" | "DOS PAGAMENTOS" | etc
  text: string;               // texto integral (imutável)
  is_immutable: boolean;      // true para as 49 base
}
```

**Seções das 49 cláusulas:**
- Cláusulas 1-2: Do Objeto do Contrato
- Cláusulas 3-8: Dos Valores e Forma de Pagamento
- Cláusulas 9-13: Das Condições Gerais
- Cláusulas 14-16: Das Responsabilidades do Contratante
- Cláusulas 17-20: Dos Cancelamentos e Alterações
- Cláusulas 21-28: Das Condições Específicas (Bagagem, Furtos, Câmbio)
- Cláusulas 29-32: Das Responsabilidades da Contratada
- Cláusulas 33-44: Do Transporte Aéreo
- Cláusulas 45-48: Da Rescisão e Cancelamento
- Cláusula 49: Do Foro (Chapecó/SC)

**Dados pré-preenchidos (template Excelência Tour):**
- Razão Social: EVELLYN DOS SANTOS AGENCIA DE VIAGENS
- CNPJ: 36.620.341/0001-07
- Endereço: Av. Nereu Ramos, 576 E, Centro, Chapecó/SC, CEP 89801-020
- Foro: Comarca de Chapecó/SC

> Outras agências que usem a plataforma devem preencher seus próprios dados no Brand Kit, e o template de contrato usa esses dados.

### 29.2 Regras de Negócio Críticas

```
R01: Proposta aceita → cria Trip automaticamente
R02: Trip criada → ativa checklist de embarque automático
R03: Contrato assinado → imutável (trigger no banco)
R04: Trip com departure_date passada → arquivada automaticamente
R05: Voucher gerado → NUNCA incluir contatos de operadoras B2B
R06: OCR de cotação → filtrar valores de comissão e ADM
R07: Magic link de passageiro → expira em 30 dias
R08: Magic link de contrato → expira em 90 dias
R09: Parcelas de pagamento → gerar notificação 7d, 3d, 1d antes
R10: Alteração de voo detectada → criar ticket automaticamente
R11: Aniversariante em 15 dias → trigger de automação WhatsApp
R12: Check-in da cia aérea abre 48h antes → criar alerta no embarque
R13: Cláusulas pétreas → nunca aparecer opção de edição/exclusão
R14: Cofre de contratos → sem opção de delete
R15: Audit log → append-only, sem delete, sem update
```

## 30. Segurança, LGPD e Compliance

### 30.1 Autenticação e Autorização

- Supabase Auth com JWT
- MFA disponível (TOTP)
- Magic links para clientes e passageiros (expiração configurável)
- Sessões com refresh automático
- Logout forçado em caso de violação

### 30.2 Papéis e Permissões

| Papel | Acesso |
|---|---|
| `super_admin` | Tudo, incluindo dados de outras agências |
| `agency_admin` | Tudo da própria agência |
| `agent` | CRM, propostas, viagens, suporte (sem financeiro completo) |
| `agent_viewer` | Somente leitura |
| `client` | Portal do cliente (próprios dados) |
| `passenger` | Formulário magic link (preenchimento de dados próprios) |

### 30.3 LGPD

**Dados coletados e finalidade:**
- Dados pessoais de clientes → execução de serviço turístico
- Dados de passageiros → obrigação legal (emissão de passagens)
- Dados de navegação → analytics interno

**Direitos do titular:**
- Acesso → portal do cliente mostra todos os dados
- Retificação → cliente pode editar dados no perfil
- Exclusão → solicitação via ticket (dados de contratos: 5 anos mínimo por lei)
- Portabilidade → exportação em JSON/PDF

**Consentimento:**
- Checkbox obrigatório no onboarding da agência
- Checkbox obrigatório no cadastro do cliente
- Timestamp de aceite registrado no audit_log

### 30.4 Validade Jurídica dos Contratos

Os contratos eletrônicos do TravelOS seguem os requisitos da **MP 2.200-2/2001** (ICP-Brasil) e do **Decreto 10.543/2020**:

- Assinatura eletrônica qualificada com:
  - Identificação do signatário (nome + CPF)
  - Autenticação por email/SMS
  - Registro de IP, user-agent, geolocalização
  - Timestamp certificado
  - Hash do documento (SHA-256)
  - Selfie no momento da assinatura

- Página de verificação pública: `/verify/:certificate_serial`

### 30.5 Armazenamento

- Contratos assinados: bucket Supabase Storage com policy `no_delete`
- Retenção mínima: 5 anos (por imposição legal)
- Backup automático diário (Supabase)
- Criptografia em repouso e em trânsito

## 33. Storyboards e Fluxos End-to-End

### Fluxo 1: Lead → Contrato Assinado

```
1. [Agente - CRM] Recebe lead pelo formulário do Instagram
   → Lead aparece na coluna "Novo Lead" do Kanban
   
2. [Agente - CRM] Abre o card → vê dados preenchidos pelo form
   → Edita preferências, adiciona nota, move para "Em Contato"
   
3. [Agente - Proposta] Cria nova proposta vinculada ao lead
   → Faz upload da cotação da operadora (PDF)
   → IA extrai todos os dados e preenche o CMS
   → Revisa, ajusta preços, seleciona template
   → Gera imagem de capa com IA
   → Preview do A4 aprovado
   
4. [Agente - Proposta] Envia link da proposta via WhatsApp
   → Move lead para "Proposta Enviada"
   
5. [Cliente - Web] Recebe link, abre a proposta no celular
   → Visualiza: destino, voos, hotel, passeios, preços
   → Clica "Aceitar Proposta"
   
6. [Sistema] Registra aceite com IP + dispositivo
   → Cria Trip automaticamente
   → Notifica agente
   → Lead movido para "Ganho"
   
7. [Agente - Contrato] Gera contrato da Trip
   → Visualiza preview do A4
   → Envia magic link para assinatura
   
8. [Cliente - Web] Recebe link do contrato
   → Lê as cláusulas na íntegra
   → Preenche dados pessoais
   → Tira selfie com câmera
   → Desenha assinatura no canvas
   → Confirma geolocalização
   → Clica "Assinar"
   
9. [Sistema] Gera certificado digital
   → Salva PDF imutável no Storage
   → Envia PDF por email para cliente e agência
   → Trip status → 'contracted'
```

### Fluxo 2: Embarque Completo

```
30 dias antes do embarque:
1. [Sistema] Trigger automático → 
   - Envia voucher ao cliente via WhatsApp
   - Abre item do checklist "Enviar voucher"
   - Cria notificação para o agente

48h antes:
2. [Sistema] → 
   - Alerta de check-in para o agente
   - Abre item "Fazer check-in online"
   
3. [Agente - Embarques] Abre card do embarque
   → Clica "Smart Check-in" → abre site da cia com PNR
   → Faz check-in e obtém cartões de embarque
   → Faz upload dos cartões no sistema
   → Marca item como concluído
   → Envia cartões de embarque ao cliente via WhatsApp

Dia do embarque:
4. [Agente - Embarques] Monitora voo em tempo real
   → Se alteração detectada → ticket aberto automaticamente
   → Caso tudo OK → marca "Voo partiu" no checklist
   
5. [Sistema] 
   → 7 dias após retorno → move viagem para "Encerrada"
   → Solicita avaliação do cliente
```

### Fluxo 3: Voucher via OCR

```
1. [Agente - Vouchers] Upload do PDF da operadora (ex: FRT)
2. [Gemini] OCR: extrai texto completo do PDF
3. [Gemini] Estrutura: 
   - Identifica passageiros (filtra nome da agência)
   - Extrai voos (datas, horários, CIA, PNR)
   - Extrai hotel (check-in/out, regime)
   - Extrai transfer (fornecedor LOCAL, telefone LOCAL)
   - Extrai seguro (apólice, central 24h)
   - IGNORA: contatos FRT, taxas B2B, regras de cancelamento
4. [Sistema] Preenche schema do voucher automaticamente
5. [Agente] Revisa e ajusta dados se necessário
6. [Sistema] Gera 3 documentos (Story 9:16):
   - Voucher completo (capa + voos + hotel + transfer + passeios + seguro)
   - Guia de Embarque (documentação + bagagem + chimarrão)
   - Guia de Emergência (contatos locais + SAMU 192 + etc)
7. [Agente] Gera mensagem WhatsApp personalizada
8. [Agente] Envia tudo para o cliente
```

## NOTAS FINAIS PARA O LOVABLE

```
1. NUNCA criar popups/modais — sempre SheetPages (drawer lateral ou full page)
2. NUNCA usar gradientes, sombras decorativas ou cores neon
3. SEMPRE usar os tokens CSS do design.tokens.css
4. SEMPRE usar lucide-react para ícones
5. Scroll: overflow: auto + scrollbar-width: none em todo lugar
6. Documentos A4: width fixa em CSS mm, NÃO responsivo
7. Canvas de documentos: transform: scale() calculado via JS, nunca CSS responsivo
8. Contratos assinados: NENHUMA UI de edição/delete deve existir
9. Cláusulas pétreas (1-49): NUNCA renderizar controles de edição sobre elas
10. Todos os CRUDs: SheetPage com botão [X] no canto superior direito
11. Todas as listas: scroll interno sem barra visível
12. Sidebar: colapsável com CSS transition, não com JS condicional
13. Brand color: sempre via CSS var(--color-brand), nunca hardcoded
14. IDs e localizadores: sempre em font-mono
15. A IA chat (coluna 2) é persistente entre navegações via Zustand
```
